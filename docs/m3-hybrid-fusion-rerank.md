# M3.4 双路检索融合与轻量 Rerank

## 这一阶段解决了什么问题

在上一阶段里，新闻助手已经具备了两条检索链路：

- 本地新闻检索
- Tavily Web Search

但当时的结果融合方式仍然比较粗糙，本质上是：

- 本地来源取前 3 条
- Web 来源取前 3 条
- 然后直接拼接返回

这种做法虽然能跑通功能，但还不够像企业项目。它的主要问题是：

1. 没有统一排序标准  
   本地检索分数和 Tavily 分数不是一个量纲，直接拼接后很难比较“谁更应该排前面”。

2. 没有真正的融合  
   它只是把两组结果放在一起，而不是判断哪些来源更相关、更新鲜、更值得作为回答证据。

3. 缺少来源平衡  
   如果单一路径结果过强，容易导致另一条路径完全没有展示空间，不利于用户理解“本地来源”和“外部来源”如何互补。

4. 还没有 rerank  
   这意味着当前回答虽然已经 grounded，但在检索融合层还没有进入更完整的 hybrid retrieval 设计。

因此这一阶段的目标不是直接上向量库，而是先把“双路检索”升级成“轻量融合 + 初步 rerank”。

## 为什么先做轻量 Rerank

这一阶段没有直接引入 Qdrant 或正式 reranker 模型，而是先做一个规则驱动的轻量 rerank。原因有三个：

1. 让多检索源先有统一排序标准  
   在本地 lexical baseline 和 Tavily 都已经接入的前提下，先解决“怎么融合”比立刻上向量库更关键。

2. 保持可解释性  
   当前项目还在为面试服务。规则式 rerank 很适合作为一个中间演进阶段，因为它可以明确讲清：
   - 为什么这一条来源排前面
   - 为什么本地新闻和 Web 来源会一起进入最终证据集
   - 为什么时间更近、标题更命中、跨源互证的来源权重更高

3. 为后续 Qdrant / LangGraph 留稳定接口  
   这一阶段把“融合与排序”独立成单独模块。后面无论你把本地 lexical retrieval 替换成 BM25、Qdrant dense retrieval，还是引入真正的 reranker，这层接口都可以保留。

## 实现原理

### 1. 新增独立的融合服务层

新增：

- `backend/services/retrieval_fusion_service.py`

这一层负责：

- 多源结果统一打分
- 去重
- 平衡选源
- 轻量 rerank
- 置信度估计

这样 `news_agent_service.py` 继续只做编排，不再自己堆融合规则。

### 2. 统一分数尺度

本地来源和 Tavily 来源原始分数不在一个量纲：

- 本地 lexical retrieval 分数可能是 `3~10+`
- Tavily 分数通常是 `0~1`

因此先做归一化：

- 本地来源：按上限截断后映射到 `0~1`
- Web 来源：直接按 `0~1` 使用

这样不同来源才能进入统一排序。

### 3. 轻量 Rerank 特征

当前融合分数由这几类信号组成：

1. 基础检索分数  
   表示原检索链路本身已经认为它相关。

2. 问题词项重叠  
   看用户问题中的关键词、标题词、片段词在来源标题和摘要里是否命中。

3. 字符 bigram 重叠  
   用轻量级字符片段相似度补 lexical retrieval 的不足，增强对近义表达和短语片段的容忍度。

4. 时间新鲜度  
   根据 `all / 24h / 7d` 不同时间范围，对更近的新闻给更高加分。

5. 来源先验  
   本地新闻在当前项目里是结构化业务数据，来源可追踪，因此默认会有小幅先验优势；但 Web 来源仍然保留补充价值。

6. 跨源互证 bonus  
   如果本地来源和 Web 来源在标题/摘要里共享较多关键项，会额外加分。  
   这代表两条链路都在支持相似事实，更适合作为 grounded answer 的证据。

### 4. 平衡选源策略

Rerank 之后不会单纯取分数最高的前 N 条，而是做了一层平衡：

- 如果本地和 Web 都有可用结果，尽量保留两侧的代表来源
- 限制单一来源类型过度占满最终证据集
- 对 Web 来源增加域名多样性限制，避免同一个站点连续占太多位

这样做的目的是：

- 提高证据集的可解释性
- 让用户看得出“本地新闻平台”和“外部 Web 搜索”是如何协同工作的
- 为后面真正的 hybrid retrieval / rerank 铺路

### 5. 统一置信度估计

这一阶段也把置信度估计改成基于融合结果，而不是简单看“本地有没有命中、Web 有没有命中”。

当前置信度主要考虑：

- 融合后 Top1 的综合分
- Top3 的平均综合分
- 是否同时存在本地和 Web 来源
- 最终证据集数量

这让 `confidence` 更接近“最终回答证据质量”，而不是某一条检索链路的局部分数。

## 前端改动

AI 页没有改接口契约，但展示上做了一个小调整：

- 来源卡片里的分数字样，从“匹配”改成了“综合”

原因是当前返回给前端的 `score` 已经不再只是某一条检索链路的原始得分，而是融合后的综合排序分数。

这能让前端展示和后端语义保持一致。

## 这一步的意义

这一阶段的价值不是“最终版检索系统已经完成”，而是：

1. 双路检索从“可用”升级到“更像真实系统”
2. 引入了初步 rerank 思维，但仍保持可解释
3. 把融合逻辑独立成单独模块，为后续升级留接口
4. 给面试中的方法演进提供了非常自然的一步

你后面可以这样表述这一步：

“在接入本地新闻检索和 Tavily 之后，我没有继续停留在简单拼接结果，而是补了一层轻量融合与初步 rerank。它先统一不同检索源的分数尺度，再综合考虑关键词重叠、时间新鲜度、来源先验和跨源互证，对最终证据集做平衡选源。这一步还不是最终的向量检索和模型 reranker，但它把系统从‘多来源可用’推进到了‘多来源可排序、可解释、可扩展’。” 

## 手动测试

1. 启动前后端，确保本地检索和 Tavily 都可用
2. 打开 AI 页，确认顶部状态正常
3. 提问一个本地库和外部搜索都可能命中的问题
4. 观察来源卡片顺序是否比之前更稳定，且本地来源与 Web 来源能共同出现
5. 看来源卡片中的分数字样，应该显示为“综合”
6. 切换 `近 24 小时 / 近 7 天`，观察更近的新闻是否更容易排前
7. 提问一个本地和 Web 都高度相关的问题，观察回答是否更容易出现跨源支撑
