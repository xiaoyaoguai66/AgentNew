# M2.4：AI 页面升级为新闻助手页

## 目标

这一阶段不直接接入 RAG、工作流或来源引用，而是先把 AI 页从“普通聊天页”升级成“面向新闻场景的助手页”。

核心目标：

- 让 AI 页有明确的新闻场景定位
- 补齐新闻化提问模板和交互骨架
- 为后续 `M3` 新闻 Agent 后端能力预留展示结构

## 本次改动

### 1. 新增 AI API 封装

新增：

- `frontend/src/api/ai.js`

作用：

- 统一处理 `/api/ai/chat`
- 与前面新闻 API 的组织方式保持一致

### 2. 重写 AI 页面

重写：

- `frontend/src/views/AIChat.vue`

新页面包含：

- 顶部新闻助手 hero 区
- 回答模式切换
- 时间范围筛选
- 主题筛选
- 快捷问题卡片
- 会话记录区
- 固定底部输入框
- 新对话按钮

### 3. 增加新闻化提示词编排

虽然当前后端还是通用模型调用，但前端发送前会自动补齐：

- 任务模式
- 时间范围
- 关注主题
- 回答结构要求
- 新闻场景边界提醒

这样做的作用是：

- 让当前模型输出更贴近“新闻助手”
- 在不改后端接口的前提下，先把用户体验调成新闻 Agent 的方向

### 4. 保持能力边界透明

页面明确说明：

- 当前还是通用模型驱动
- 下一阶段才会上 RAG、引用来源、置信度

这很重要，因为它避免了前端“假装已经是 Agent”的问题，便于后续逐步演进。

## 为什么这一步重要

### 1. 先把产品形态做对

后面即使你把后端 Agent 做得很强，如果前端还是普通聊天框，项目呈现出来的完成度也不够。

### 2. 为 M3 做承接

这一步已经把后续 Agent 页最重要的交互位置预留好了：

- 任务模式
- 主题筛选
- 快捷提问
- 会话区
- 新对话入口

后续接 RAG 和来源引用时，不需要推翻页面结构。

### 3. 面试里更容易说明演进路径

你可以明确讲：

1. M2 阶段先把 AI 页升级为新闻助手页
2. 通过提示词工程把输出尽量拉到新闻场景
3. M3 再把后端升级成真正的新闻 Agent

这会让项目路线更清楚，也更像真实迭代过程。

## 手工测试建议

### 1. 验证页面结构

1. 打开 AI 页
2. 观察是否出现：
   - 新闻助手 hero 区
   - 回答模式
   - 时间范围
   - 关注主题
   - 快捷问题区

预期：

- 页面不再只是一个聊天框，而是明显更像新闻助手产品页

### 2. 验证快捷问题

1. 点击任意快捷问题卡片
2. 观察是否直接发起请求并生成回答

预期：

- 会话区出现用户消息和助手回答

### 3. 验证筛选对提问的影响

1. 切换不同回答模式、时间范围和主题
2. 输入相似问题重复提问

预期：

- 回答风格会明显不同
- 例如“事件梳理”和“对比分析”模式的结构不同

### 4. 验证新对话

1. 发送几轮消息
2. 点击右上角“新对话”

预期：

- 会话清空并恢复到欢迎消息

### 5. 验证异常场景

1. 暂时让后端 AI 接口不可用或移除 API Key
2. 再次发送问题

预期：

- 页面不会白屏
- 会显示错误消息或 toast

## 面试可讲的点

- 为什么先做新闻化提示词编排，而不是立刻上全量 Agent
- 为什么要把回答模式、时间范围和主题筛选前置到 UI
- 为什么要明确区分“通用模型模式”和“后续 RAG Agent 模式”
- 如何通过前端交互设计为后续 Agent 工作流预留结构
