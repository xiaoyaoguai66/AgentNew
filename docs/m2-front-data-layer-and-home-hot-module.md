# M2.1：前端数据层整理与首页热门快读模块

## 目标

这一阶段不直接大面积重做所有前端页面，而是先处理两个对项目长期价值最高的点：

- 把新闻前端的数据请求从“页面和 store 里到处直接写 axios”整理成统一 API 层
- 把后端刚完成的热门新闻能力接到首页，形成可见、可演示、可测试的移动端功能

这样做的原因很实际：

- 后续首页、详情页、AI 页继续升级时，不会重复堆请求代码
- 可以先验证后端 Redis 热榜能力是否真正对前端有业务价值
- 项目在 GitHub 和面试里更容易讲清楚“前后端如何协同”

## 本次改动

### 1. 新增统一新闻 API 层

新增：

- `frontend/src/api/http.js`
- `frontend/src/api/news.js`

职责：

- `http.js`
  - 统一管理前端请求基础地址和超时
  - 统一处理接口错误提示
- `news.js`
  - 封装新闻分类、列表、详情、热榜请求
  - 对后端返回数据做 normalize，给页面稳定的前端数据结构

统一后的好处：

- 页面和 store 不再直接关心 `response.data.code === 200`
- 后端字段别名变化时，只需要在 adapter 层处理
- 后续接 Agent 来源列表、推荐问题等功能时，模式可以直接复用

### 2. 重写新闻 store

重写：

- `frontend/src/store/modules/news.js`

新增能力：

- `hotNews`
- `hotLoading`
- `listError`
- `detailError`
- 分页状态 `page / pageSize / total`

保留原有调用习惯：

- `getCategories`
- `changeCategory`
- `getNewsList`
- `getNewsDetail`

这样其他页面不会立刻被破坏，同时 store 内部结构更清晰。

### 3. 首页新增热门快读模块

重写：

- `frontend/src/views/Home.vue`

首页现在新增：

- 顶部品牌化 hero 区块
- 热门快读模块
- 刷新热门按钮
- 分类新闻流错误态和空态

热门快读使用：

- `GET /api/news/hot`

当切换分类时：

- 热门快读和列表一起刷新

因此首页能够直接体现：

- Redis 热榜
- 分类切换联动
- 前后端协作后的真实业务价值

### 4. 分类页同步升级

重写：

- `frontend/src/views/Category.vue`

作用：

- 直接加载分类
- 进入首页时通过 `query.categoryId` 驱动当前分类

### 5. 新闻卡片升级

重写：

- `frontend/src/components/NewsItem.vue`

变化：

- 更适合移动端阅读的信息层级
- 统一展示发布时间和阅读量格式
- 无图时提供占位样式，避免布局塌陷

### 6. 新增新闻展示工具函数

新增：

封装：

- 阅读量格式化

## 为什么这一阶段重要

### 1. 这是“企业级前端”的基础，不是单纯换皮

真正的企业级前端，第一步通常不是改颜色，而是：

- 数据流是否统一
- 页面是否能复用稳定的数据模型
- 错误态、空态、加载态是否有明确处理

### 2. 让后端热榜能力真正落地

前面做 Redis 聚合和热榜，如果前端没有消费，这个能力就只能停留在接口层。

这一步让项目形成了完整链路：

- 后端 Redis 热度累计
- 热榜接口输出
- 首页热门快读可视化展示

### 3. 为后续 AI 页和详情页升级打底

后续做新闻 Agent 时，前端也会需要统一的：

- 请求封装
- 状态管理
- 引用来源展示模型

这一步已经把方向定好了。

## 手工测试建议

### 1. 验证首页热门快读

1. 启动前后端
2. 打开首页
3. 观察顶部是否出现“热门快读”模块
4. 热门区域应显示当前分类下的热门新闻

预期：

- 能看到 1 条主热点 + 若干条次级热点
- 点击热点可进入详情页

### 2. 验证分类切换联动

1. 在首页切换分类 tab
2. 观察新闻列表和热门快读是否同时变化

预期：

- 切换分类后，列表重新加载
- 热门快读刷新为对应分类的热榜数据

### 3. 验证分类页跳转

1. 进入“全部分类”页
2. 点击任意分类
3. 应跳回首页并自动切到对应分类

预期：

- 首页 tab 正确切换
- 新闻列表和热门快读都对应新分类

### 4. 验证详情跳转

1. 点击新闻列表中的任意新闻
2. 点击热门快读中的任意新闻

预期：

- 都能进入详情页
- 详情页继续走原有后端详情接口

### 5. 验证热榜与后端 Redis 联动

1. 多次访问某条新闻详情
2. 返回首页
3. 观察对应分类下的热门快读排序变化

预期：

- 高频访问的新闻更容易进入热门区域靠前位置

## 面试可讲的点

- 为什么前端要增加 API adapter 层
- 为什么 store 里要维护 `hotNews` 和分页状态
- 首页热门模块如何消费 Redis 热榜接口
- 如何把后端缓存与前端体验结合起来，而不是只做后端优化
