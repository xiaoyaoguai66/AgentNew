# M3.2 本地新闻检索与 Grounded Answer

## 这一阶段解决了什么

上一阶段只是把 prompt 编排收回后端，但新闻助手本质上仍然是通用模型调用，所以会有两个明显问题：

1. 不能真正回答你项目里“具体新闻”的问题。
2. 没有实时联网能力，也没有来源回传。

这一阶段先补上第一块能力：让新闻助手先从本地新闻库里检索候选新闻，再基于这些候选来源生成回答。

## 当前能力边界

现在的新闻助手具备：

- 本地新闻库检索
- 后端统一 prompt 编排
- 基于来源的回答生成
- 返回来源列表
- 返回置信度
- 明确告诉前端当前是否开启检索和联网搜索

现在仍然不具备：

- 实时联网 web search
- Tavily 接入
- 向量检索
- RAG rerank
- LangGraph 工作流

也就是说，这一阶段是“本地检索骨架”，不是最终版 Agent。

## 本次改动

### 1. AI 接口返回来源与置信度

`/api/ai/chat` 现在除了回复文本，还会返回：

- `sources`
- `confidence`
- `retrievalEnabled`
- `webSearchEnabled`

这样前端可以知道：

- 当前回答是不是基于检索
- 当前是否已经具备实时联网
- 回答对应了哪些新闻来源

### 2. 新增本地新闻检索服务

新增 `backend/services/news_retrieval_service.py`。

当前检索方式是轻量版本：

- 根据前端传入的主题和时间范围做初筛
- 从 MySQL 新闻表里取最近候选新闻
- 对问题和候选新闻做关键词与字符片段匹配打分
- 取分数最高的几条作为证据来源

这不是最终的向量检索方案，但它能先把“回答有来源”这件事建立起来。

### 3. Grounded Prompt

Prompt 现在不只是“回答问题”，而是明确要求：

- 只能基于给定新闻证据回答
- 不要伪造来源
- 不要声称自己已经联网
- 证据不足时明确表示不确定

这一步是后续做幻觉控制的基础。

### 4. 前端显示来源卡片

AI 页现在会在每条助手消息下方展示：

- 可信度
- 来源卡片
- 来源时间
- 匹配分数

点击来源可以直接跳到新闻详情页。

## 为什么现在先不接 Tavily

因为在你的项目路线里，更合理的顺序是：

1. 先把本地新闻检索闭环跑通
2. 再把外部 web search 作为第二检索源接进来
3. 最后做本地新闻库 + web search + 向量检索的混合检索

如果一开始就接 Tavily，但没有把本地检索和来源结构设计好，后面反而会推倒重来。

## 后续 Tavily 接入的落点

这一阶段接口里已经留出了：

- `retrievalEnabled`
- `webSearchEnabled`

后面接 Tavily 时，建议新增一层统一检索编排：

- local news retrieval
- vector retrieval
- Tavily web search
- merge + rerank

这样前端不用再改协议。

## 手动测试

1. 启动前后端。
2. 打开新闻助手页，输入与你本地新闻库内容相关的问题。
3. 确认回答下方会显示来源卡片。
4. 点击来源卡片，确认能跳转到对应新闻详情页。
5. 切换时间范围和主题，观察来源是否发生变化。
6. 问一个明显超出本地新闻库的问题，确认助手不会硬编，而是提示本地新闻证据不足。
7. 观察页面顶部状态，确认：
   - 检索为已开启
   - 联网为未开启
