# M2.2：新闻详情页阅读体验升级

## 目标

这一阶段的核心不是新增后端接口，而是把新闻详情页从“能展示内容”提升到“适合作为移动端新闻 App 主阅读页”的层级。

具体目标：

- 更强的移动端阅读体验
- 明确的加载态、错误态、空态
- 点击相关推荐时同页稳定切换，不再依赖组件销毁重建
- 收藏与浏览历史逻辑继续保留
- 为后续 Agent 来源展示页留出结构空间

## 本次改动

### 1. 重构详情页状态流

重写：

- `frontend/src/views/NewsDetail.vue`

新页面结构包含：

- 顶部导航
- 文章信息头图卡片
- 正文阅读卡片
- Agent 来源延展说明卡片
- 相关推荐列表

相比旧版本，详情页不再只是“标题 + 正文 + 相关推荐”的简单拼接，而是更接近真实新闻 App 的阅读结构。

### 2. 增加加载态、错误态、空态

新增状态处理：

- `detailLoading`
- `detailError`

页面行为：

- 请求中显示骨架屏
- 请求失败显示错误卡片和重试按钮
- 没有内容时显示空态

这样做的价值是：

- 用户不会看到空白页
- 路由快速切换时状态更稳定
- 项目在演示时更像真实产品，而不是开发态页面

### 3. 修正详情页路由切换逻辑

旧详情页有一个隐性问题：

- 点击相关推荐进入另一条新闻时，仍然是同一个路由组件实例
- 如果不监听路由参数变化，可能不会重新加载详情

现在新增了：

- 对 `route.params.id` 的监听

这样在详情页内部继续跳详情页时，会重新拉取数据并滚动回顶部。

### 4. 优化 store 的详情请求行为

调整：

- `frontend/src/store/modules/news.js`

新增：

- `clearNewsDetail`
- 请求详情前主动清空旧详情数据

这样可以避免：

- 新详情加载时短暂显示上一条新闻内容

### 5. 收藏与浏览历史链路保留

详情页仍然保留：

- 登录后收藏
- 进入详情时记录浏览历史
- 打开详情后同步收藏状态

只是把原来比较杂乱的逻辑收口到了更清晰的加载流程中。

## 为什么这一步重要

### 1. 详情页是新闻类产品的主价值页

首页可以展示内容流，但真正决定产品阅读体验的是详情页。

如果详情页排版松散、状态混乱，整个项目看起来就更像 demo。

### 2. 这是后续 Agent 引用展示页的天然入口

后面做新闻 Agent 时，最自然的展示链路就是：

- Agent 给出回答
- 列出引用来源
- 点击来源跳到新闻详情页

所以详情页不仅是阅读页，也会成为“证据来源页”。

### 3. 面试里更容易讲“产品思维”

你可以明确说：

- 首页解决发现内容
- 详情页解决深度阅读
- Agent 页解决问答与信息聚合

这会让项目从“技术堆栈展示”升级成“有业务链路的应用”。

## 手工测试建议

### 1. 验证详情页加载态

1. 打开任意新闻详情页
2. 观察页面在请求过程中是否先显示骨架占位

预期：

- 不会先闪出旧内容
- 请求完成后再渲染新文章

### 2. 验证相关推荐跳转稳定性

1. 打开任意新闻详情页
2. 点击下方相关推荐中的另一条新闻

预期：

- 页面会切换到新文章
- 自动滚动回顶部
- 标题、正文、阅读量、相关推荐都同步刷新

### 3. 验证收藏逻辑

1. 登录账号
2. 进入新闻详情页
3. 点击收藏按钮，再点击取消

预期：

- 收藏状态可切换
- 有 toast 提示
- 再进入收藏页时状态一致

### 4. 验证浏览历史记录

1. 登录账号
2. 打开若干条新闻详情
3. 进入历史页

预期：

- 浏览过的新闻被记录下来

### 5. 验证错误态

1. 临时关闭后端或手动输入不存在的新闻 ID
2. 访问详情页

预期：

- 页面显示错误或空态，而不是直接白屏

## 面试可讲的点

- 为什么详情页要监听路由参数变化
- 为什么请求详情前要先清空旧数据
- 为什么加载态和错误态是“企业级前端”的一部分
- 为什么新闻详情页未来适合承接 Agent 的来源引用展示
