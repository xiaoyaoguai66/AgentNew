# M3.3 Tavily 双路检索接入

## 这一阶段解决了什么

上一阶段已经完成了本地新闻检索和 grounded answer，但仍然有一个明显边界：

- 只能回答项目本地新闻库里已有的数据
- 对最新外部新闻和突发变化覆盖不足

因此这一阶段新增了第二检索源：Tavily Web Search。

## 这次为什么这样做

这一步不是为了“把本地检索推翻重做”，而是为了把检索能力升级成双路：

1. 本地新闻检索  
   负责项目自有新闻数据、可解释来源和稳定基础问答。

2. Tavily Web Search  
   负责最新网页信息、外部补充来源和本地库未覆盖的内容。

这样做的原因是：

- 本地库更稳定、更可控
- Web 搜索更实时
- 两者结合更接近真实新闻 Agent 场景

## 实现原理

### 1. 后端支持 Tavily 配置

新增环境变量：

- `TAVILY_API_KEY`
- `TAVILY_BASE_URL`
- `TAVILY_SEARCH_DEPTH`
- `TAVILY_MAX_RESULTS`

如果没有配置 `TAVILY_API_KEY`，后端会自动降级成“仅本地检索”。

注意：

- 运行时读取的是项目根目录 `.env`
- 也兼容 `backend/.env`
- `.env.example` 只是模板，不会被应用直接加载

如果只把 key 写进 `.env.example`，重启后端也不会生效。

### 2. Tavily 被设计成可开可关的第二检索源

新增 `backend/services/tavily_service.py`，负责：

- 组装 Tavily 搜索请求
- 根据主题映射搜索 topic
- 根据时间范围映射 `time_range`
- 解析 Tavily 返回结果
- 转成统一的 `AiSourceItem`

后面这个模块又补了一层中文 query 适配：

- 如果用户问题包含中文，系统会先执行一次 `Web Query Rewrite`
- 把中文问题改写成更适合 Tavily 的英文搜索词
- 再把改写后的 query 发给 Tavily

这是因为实际联调里发现，部分中文 query 会被 Tavily 直接判定为 `Query is invalid`。

### 3. 双路检索编排

在 `news_agent_service.py` 里，现在会并行执行：

- 本地新闻检索
- Tavily 搜索

然后做：

- 来源合并
- 来源去重
- 统一排序与截断
- grounded answer 生成

### 4. 降级策略

这是这一阶段很重要的一部分。

如果 Tavily：

- 没配 API key
- 调用失败
- 网络不可用
- 返回非法数据

则不会影响整个助手流程，系统仍然会正常回退到本地新闻检索。

这样做是为了避免把 Web 搜索变成单点依赖。

## 为什么没有直接切成“纯 Web 搜索”

因为你的项目目标不是做一个普通聊天页，而是做一个“新闻平台 + 新闻 Agent”。

如果完全依赖 Web 搜索，会失去：

- 本地新闻资产的价值
- 本地业务数据的可控性
- 项目内部数据链路的可解释性

所以这一步的正确方向是：

本地检索作为第一检索层  
Web 搜索作为第二检索层

## 为什么 Tavily 参数这样选

当前默认策略是：

- `search_depth=basic`
- `max_results=3`

这么做是出于工程权衡：

1. `basic` 更适合作为第一版接入  
   延迟和成本都更容易控制。

2. `max_results=3` 足够作为补充来源  
   当前目标不是把网页塞满上下文，而是补齐“本地新闻库之外的实时证据”。

3. 先打通接口，再做 rerank  
   这一阶段先把双路检索骨架跑通，后面再继续做结果融合优化。

## 前端改动

前端新闻助手页现在可以区分：

- 本地新闻来源
- Web 搜索来源

其中：

- 本地来源点击后跳转新闻详情页
- Web 来源点击后打开外部链接

同时页面顶部也会展示：

- 当前是否开启双路检索
- 当前是否开启 Tavily

这一部分后来补了两次状态同步修复：

- 第一次修复：之前页面顶部状态只会在首轮 `/api/ai/chat` 返回后更新，现在新增 `/api/ai/status`，页面加载时就会先拉取当前能力状态
- 第二次修复：根组件对 `AIChat` 启用了 `keep-alive`，这意味着页面第一次挂载后再次进入时不会重新触发 `onMounted`
- 因此又把状态同步补到了 `onActivated`，保证用户切到别的 Tab、重启后端、调整 `.env` 后再回到 AI 页时，顶部状态会重新拉取，而不是继续显示旧值

这样配置好 `TAVILY_API_KEY` 并重启后端后，用户一打开 AI 页就能看到真实状态，而不是必须先发起一次对话。

## 手动测试

### 情况一：未配置 Tavily

1. 不配置 `TAVILY_API_KEY`
2. 启动前后端
3. 打开新闻助手页
4. 提问后应仍可正常回答
5. 页面顶部应显示 Tavily 未开启
6. 来源应主要来自本地新闻

### 情况二：已配置 Tavily

1. 在 `.env` 中配置 `TAVILY_API_KEY`
2. 重启后端
3. 打开新闻助手页
4. 不发送消息时，顶部状态就应显示 Tavily 已开启
5. 切换到底部其他 Tab 后再返回 AI 页，顶部状态仍应保持最新，不需要先发消息
6. 提问一个本地新闻库可能不足以覆盖的问题
7. 如果 Tavily 命中，应看到 Web 搜索来源卡片
8. 点击 Web 来源，应打开外部链接

### 情况三：Tavily 故障降级

1. 配置一个错误的 Tavily key 或断网
2. 重新提问
3. 系统不应直接报错崩掉
4. 应退回到本地检索回答

## 这一步的意义

这一阶段的价值不是“已经做完最终检索系统”，而是：

- 把本地数据和外部实时数据纳入同一套来源协议
- 为后续混合检索和 rerank 铺路
- 为后续 LangGraph 工作流保留统一检索入口
