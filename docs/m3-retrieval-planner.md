# M3.5 检索规划器：从统一双路检索到 local-first / hybrid / web-first

## 这一阶段要解决什么问题

在 `M3.4` 里，项目已经完成了这些能力：

- 本地新闻检索 baseline
- Tavily Web Search
- 查询改写（中文问题先改成英文 Web Query）
- 轻量融合与初步 rerank
- 超时治理与降级

这说明双路检索已经“能跑”，但还没有做到“按问题类型走不同策略”。

`M3.4` 之前的核心问题是：

1. 所有问题默认都走同一条双路检索思路  
   不管问题是“基于站内新闻总结一下”，还是“今天最新的卫星发射进展”，后端都会采用近似相同的检索套路。

2. Web Search 虽然接进来了，但优先级还不够可控  
   某些问题天然更适合 `web-first`，例如：
   - 今天/刚刚/最新/实时类问题
   - 显式要求“联网查一下”“官网消息”的问题  
   如果仍然默认走统一融合，用户体验和工程可解释性都不够强。

3. 工作流编排还缺一个明确的“决策节点”  
   在真正升级到 LangGraph 之前，后端至少要先具备一个显式的 `Retrieval Planner`，决定后续该走哪条检索路径。

所以 `M3.5` 的目标不是直接上向量库，而是：

**先把检索工作流拆成明确的三种计划：`local-first / hybrid / web-first`。**

---

## M3.4 完成检查

进入 `M3.5` 之前，我把 `M3.4` 的完成情况明确记录如下。

已完成：

- 本地新闻检索和 Tavily 双路接入
- 中文问题的 Web Query Rewrite
- 双源融合与轻量 rerank
- 来源卡片、可信度、来源跳转
- Tavily/LLM 超时治理
- Tavily 失败时自动降级到本地检索

未完成但已经暴露出来的缺口：

- 没有按问题类型切换检索计划
- “本地优先”和“Web 优先”还只是隐含现象，不是正式工作流节点
- 前端和响应结构里没有显式暴露“规划结果”
- 工作流还没有形成 `planner -> retrieval -> fusion -> generation` 的稳定结构

因此 `M3.4` 可以视为：

**双路检索和融合能力已经完成，但“检索规划”这一层还缺失。**

这正是 `M3.5` 的工作范围。

---

## 为什么先做检索规划，而不是立刻上向量检索

这一步的关键不是“检索精度极限优化”，而是“工作流正确性”。

在新闻 Agent 场景里，系统至少要先回答这些问题：

- 什么时候应该先用本地新闻库？
- 什么时候应该先去 Web Search？
- 什么时候要两边一起查？
- 单路结果足够强时，是否还需要继续调用另一条链路？

如果这些问题还没回答清楚，就算你立刻把本地检索升级成 Qdrant 向量检索，整体系统仍然会缺少一个明确的“决策层”。

所以 `M3.5` 的定位是：

**在检索能力继续增强之前，先把 Agent 的受控工作流补完整。**

---

## 本次实现：新增 Retrieval Planner

新增文件：

- `backend/services/retrieval_planner_service.py`

这一层负责两件事：

1. 根据问题内容、分类、时间范围、联网能力，决定当前问题应该走：
   - `local-first`
   - `hybrid`
   - `web-first`

2. 在执行阶段判断当前单路结果是否已经足够强，从而决定是否触发另一条链路作为补充或兜底。

---

## 三种检索计划的含义

### 1. local-first

适用于：

- 明确限定“本地新闻库”“站内新闻”“收藏”“历史”的问题
- 明显在引用单篇新闻上下文的问题
- Tavily 未开启时的默认兜底

执行方式：

1. 先查本地新闻库
2. 如果本地结果已经足够强，就直接进入融合/回答
3. 如果本地结果不够强，再补 Tavily

这条路径的目标是：

- 优先利用你项目自己的新闻资产
- 降低不必要的外部搜索成本
- 让“基于站内新闻总结/分析”的体验更稳定

### 2. hybrid

适用于：

- 需要分析、梳理、对比、趋势判断的问题
- 有分类/时间过滤，但又不应该完全放弃外部信息的问题
- 默认的综合型问题

执行方式：

1. 本地检索和 Web Search 并行执行
2. 结果统一进入融合和 rerank
3. 最终输出一组更平衡的证据集

这条路径的目标是：

- 同时保留本地新闻的结构化优势
- 兼顾 Web 的时效性和补充信息

### 3. web-first

适用于：

- 明显强调“今天 / 刚刚 / 最新 / 实时”的问题
- 显式要求“联网查一下”“官网消息”“网页搜索”的问题
- 对最新动态强依赖的问题

执行方式：

1. 先查 Tavily
2. 如果 Web 结果足够强，直接进入融合/回答
3. 如果 Web 结果不足，再回退补本地新闻库

这条路径的目标是：

- 让真正偏“最新信息”的问题优先获得外部信息支持
- 避免所有问题都被默认带回本地检索路径

## 规划规则是怎么实现的

当前实现不是模型自治规划，而是**规则驱动的受控 Planner**。

主要信号包括：

1. `LOCAL_SCOPE_TERMS`
   - 本地新闻
   - 新闻库
   - 收藏
   - 历史
   - 这篇新闻 / 这条新闻

2. `EXPLICIT_WEB_TERMS`
   - 联网
   - 官网
   - 官方
   - 网页
   - 搜索 / 查一下

3. `RECENCY_TERMS`
   - 最新
   - 刚刚
   - 今天
   - 实时
   - 目前
   - 最近

4. `ANALYSIS_TERMS`
   - 总结
   - 梳理
   - 分析
   - 对比
   - 趋势
   - 影响

再结合：

- `timeRange = all / 24h / 7d`
- `category = general / technology / finance / international`
- `Tavily 是否已开启`

最终做出规划决策。

这一步的工程价值在于：

- 决策逻辑可解释
- 方便后续迁移到 LangGraph 的 `Retrieval Planner` 节点
- 不需要等待向量检索落地后才能建立工作流

---

## 单路结果是否“足够强”的判断

除了先决定走哪条路，这次还补了结果强度判断：

- `is_local_result_strong(sources)`
- `is_web_result_strong(sources)`

目的不是做严格的机器学习分类，而是为受控工作流提供一个简单可解释的阈值判断。

例如：

- `local-first` 下，本地命中已经很强，就不必一定再调用 Tavily
- `web-first` 下，Web 已经有足够强的结果，也不必总是补本地检索

这样做的好处是：

- 降低不必要的调用成本
- 控制时延
- 让三种计划不只是“标签不同”，而是真的执行路径不同

---

## 与 M3.4 融合层的关系

这次不是推翻 `M3.4` 的融合逻辑，而是在其上继续前推一层。

当前链路变成：

1. `Retrieval Planner`
2. `Local Retrieval / Tavily Search`
3. `Fusion + Lightweight Rerank`
4. `Grounded Prompt`
5. `LLM Generation`
6. `Structured Response`

同时我还把 `retrieval_plan` 传入融合层，让融合排序不再完全“中立”，而会根据当前计划做轻微偏置：

- `local-first` 更偏向保留本地结果
- `web-first` 更偏向保留 Web 结果
- `hybrid` 保持较平衡的 3:3 结构

这说明现在的系统已经不是“永远同一条融合路径”，而是：

**先规划，再检索，再融合。**

## 接口与前端调试能力

这次还扩展了 AI 接口返回结构：

- `retrievalPlan`
- `plannerReason`
- `strategy`（执行路径）

前端 AI 页也同步做了调试展示：

- 顶部会显示最近一次的检索计划
- 回答区会显示：
  - 检索计划
  - 执行路径
  - 规划原因

这样做有两个好处：

1. 方便你自己调试  
   可以直接看到当前问题为什么被判成 `local-first / hybrid / web-first`

2. 方便面试展示  
   你可以直接说明：这个系统不是简单“问了就搜”，而是有显式检索规划层。

---

## 为什么这一步仍然不是“完全思考型 Agent”

这次虽然加入了 Planner，但我仍然没有把它做成完全自由的 ReAct Agent。

原因是新闻场景更适合：

- 受控工作流
- 显式规划
- 强来源约束
- 可解释的降级与回退

也就是说，`M3.5` 是在向 LangGraph 风格工作流靠近，但仍然保持：

**受控编排，而不是高自治黑盒 Agent。**

这才更适合新闻问答场景，也更适合你后面在面试里讲“为什么要控制幻觉和来源约束”。

## 手动测试建议

你可以用下面三类问题验证 Planner 是否生效：

1. `local-first`
   - “根据本地新闻库总结一下科技热点”
   - “只基于收藏里的新闻帮我做一个总结”

2. `web-first`
   - “今天最新的卫星发射计划有什么进展”
   - “帮我联网查一下今天最新的 AI 融资消息”

3. `hybrid`
   - “最近科技新闻里哪些变化最可能影响大模型行业”
   - “对比最近 7 天科技和国际新闻的变化”

预期现象：

- 回答区能看到 `retrievalPlan`
- 不同问题会触发不同的 `strategy`
- `web-first` 问题更容易优先展示 Web 结果
- `local-first` 问题会先用本地新闻库

## 这一阶段的意义

`M3.5` 的价值不是“已经完成最终形态的 Agent”，而是：

1. 把双路检索真正升级成了“可规划工作流”
2. 让 `local-first / hybrid / web-first` 成为正式的后端能力
3. 让后续 LangGraph 化有了真实工程基础
4. 让 Qdrant 混合检索接入时，有稳定的 Planner 节点可直接复用

后续升级路径就更清晰了：

1. 当前：规则驱动 Retrieval Planner
2. 下一步：Planner + 更正式的 Query Rewrite / Retrieval Planner 节点
3. 再下一步：Qdrant Hybrid Retrieval
4. 最后：LangGraph 工作流 + Verifier + 评测
